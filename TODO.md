## T-7.5で対応すべきTypeScriptエラー分類

現在、`npx tsc --noEmit`で検出されるエラーは以下の5カテゴリに分類できます：

### 1. モジュール参照エラー (60件)
- **エラーコード**: TS2307, TS2305
- **内容**: 旧パスの移行に伴うファイル参照エラー
- **例**: `Cannot find module '@/store/useSymbolStore'`
- **解決策**: インポートパスを新構造に合わせて更新
- **優先度**: 高（全体の基盤部分なので最初に対応）

### 2. 型解決エラー (55件)
- **エラーコード**: TS2571, TS2538
- **内容**: `unknown`型やnull許容型の処理が正しく行われていない
- **例**: `Object is of type 'unknown'`, `Type 'null' cannot be used as an index type`
- **解決策**: 型ガードの追加、非nullアサーション導入、オプショナルチェーニング活用
- **優先度**: 高（ランタイムエラーの原因にもなるため）

### 3. 名前衝突/再エクスポートエラー (22件)
- **エラーコード**: TS2308
- **内容**: モジュール間で同名のメンバーが再エクスポートされている
- **例**: `Module has already exported a member named 'SymbolInfoSchema'`
- **解決策**: 明示的な再エクスポート構文の使用（`export { X } from './module'`）
- **優先度**: 中（型の参照問題に比べると優先度は下がる）

### 4. 型互換性エラー (16件)
- **エラーコード**: TS2345
- **内容**: 異なる型間の変換が必要
- **例**: `Argument of type 'string' is not assignable to parameter of type ...`
- **解決策**: 型アサーション、型変換関数の追加
- **優先度**: 中（個別の関数実装に関する問題が多い）

### 5. リテラル型エラー (8件)
- **エラーコード**: TS2353, TS2678
- **内容**: リテラル文字列と列挙型の不一致
- **例**: `Type '"8h"' is not comparable to type '"1m" | "3m" | ...`
- **解決策**: 列挙型に新しい値を追加または適切な型ガードの導入
- **優先度**: 中（部分的な機能影響のみ）

## decimal.js導入分析（追加調査結果）

コードベース調査結果に基づき、数値処理に関する判断結果をまとめました。

### 現状の数値処理パターン

1. **表示フォーマット**
   - 主に`toFixed(n)`で桁数指定（価格: 8桁, 量: 4桁, パーセント: 2桁）
   - OrderBookや価格表示でゼロ末尾の除去 `toFixed(n).replace(/\.?0+$/, '')`
   - 通貨表示で`formatNumber`関数を利用（カンマ区切り、符号付き）

2. **計算精度**
   - 現状はほとんどの演算がJSのネイティブ`number`型
   - APIからのデータも`Number()`や`parseFloat()`でパース
   - チャートデータは特に精度が求められる（OHLC値）

3. **潜在的な問題箇所**
   - 取引計算（手数料、レバレッジ、損益計算など）
   - スプレッド計算（パーセント表示）
   - 暗号資産の価格表示（小数点以下多数桁）

### 導入判断

**結論: 文字列として保持 + number型演算 + 桁数固定化の組み合わせ方式を採用**

1. **decimal.jsは導入しない理由**:
   - 現段階では計算精度に関する明確な不具合が発生していない
   - 依存ライブラリ追加コストが大きい（バンドルサイズ、学習コスト）
   - 全体的な変更範囲が広すぎる（APIクライアント〜UI表示まで影響）

2. **代替アプローチ**:
   - **DB格納**: 精度が必要な値は文字列型として保存
   - **UI表示**: 定義された桁数で統一表示（通貨ごとに桁数定義）
   - **演算処理**: 必要な演算のみ精度を意識した関数を用意

3. **実装方針**:
   - `types/converters/` に数値変換関数を集約
   - 精度定義を `constants/precision.ts` として一元管理
   - UI表示用フォーマッタを統一（utils/format.ts）

4. **将来の拡張**:
   - 精度問題が顕在化した場合、特定の計算処理のみdecimal.jsを部分導入
   - または全体をBigDecimal対応にリファクタリング（大規模改修）
