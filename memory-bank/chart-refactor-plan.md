# チャートコンポーネントリファクタリング計画

## 背景と目的

チャートコンポーネントは現在、以下の自作モジュールと外部ライブラリの混合で構成されています：

- `lightweight-charts`: 主要描画ライブラリ
- `recharts`: 補助チャート用
- `store/chart/*`: チャートデータ管理
- `components/chart/*`: UIコンポーネント

このリファクタリングの目的は以下の点を改善することです：

1. チャートコンポーネントのテスタビリティ向上
2. 型システムの安定化
3. 現在の機能を保ちつつ、将来の機能追加に対応できる構造に改善

## 調査結果

### 依存関係

チャートコンポーネントへの出入りは以下のパターンが主要です：

1. `useChartDataStore`: チャートデータの状態管理
2. `useToolbarStores`: ツールバー・UI操作の状態管理
3. components/chart/sections/ChartSection: メインのコンテナコンポーネント

## リファクタリング計画

リファクタリングは以下の3フェーズに分けて実施します：

### フェーズ1：データストアの整理

**目的**: データの流れを明確にし、型安全性を向上する

- [ ] chart/data スライスの型定義の統一
- [ ] chart/config スライスの公開インターフェースの見直し
- [ ] chart/realTime のサブスクリプション管理改善

### フェーズ2：UIコンポーネントの改善

**目的**: 色トークンをTailwind化し、コンポーネントの再利用性を向上する

- [ ] DrawingToolコンポーネントのインラインスタイルをTailwind化
- [ ] カラースキームをstyles/colors.tsからtailwind.config.jsに移行
- [ ] チャートUIコンポーネントの分離と簡素化

### フェーズ3：チャート描画ロジックの整理

**目的**: 描画ロジックをReact Hooksに分離し、テスト可能性を向上

- [ ] useChartCoreのhooks化完了
- [ ] lightweight-chartsのラッパー関数の改善
- [ ] チャートイベントハンドリングの分離

## 実装手順

1. それぞれのフェーズを個別のブランチで開発
2. 各フェーズでユニットテストを追加し、既存の動作を保証
3. PRレビュー時に変更点を明確にドキュメント化

## 次のステップ

1. TS型エラーを全て解消する (現在の作業ステップ)
2. チャートのデータストアの詳細なリファクタリング計画を作成
3. フェーズ1のブランチを作成し実装開始 

## チャート用描画ツールバー

現在のチャート描画ツールバーは、使いやすさと拡張性に関して課題があります。具体的には、ツールバーUI自体とツール選択ロジックが結合しており、新しいツールやインジケーターの追加が難しくなっています。

### 課題:

1. 描画ツールとツールバーの結合度が高い
2. 描画ツールの選択とツールバー表示が密結合
3. 描画ツールの状態管理が分散している
4. UIコンポーネントがデータ構造と強く結びついている
5. カスタムツールの追加が困難

### 解決策:

1. **ツールバーのリファクタリング**:
   - ToolbarButton コンポーネント（UI）とツール選択ロジックの分離
   - 設定やスタイルをプロパティとして柔軟に受け取る構造化
   - Tailwind CSS を活用した一貫したスタイリング
   - アクセシビリティの向上とキーボード操作のサポート

2. **描画ツールの再構築**:
   - 各描画ツールをプラグイン化
   - 共通インターフェースを実装し一貫したAPI提供
   - 設定とスタイルの分離
   - 状態管理の一元化

3. **状態管理の改善**:
   - Zustand スライスによる描画ツール状態の一元管理
   - ツール設定 ⇄ UI間の双方向バインディング
   - 永続化と状態復元機能

## インジケータープラグイン

現在のインジケーターは、新しいインジケーターの追加や既存インジケーターのカスタマイズが容易ではありません。

### 課題:

1. インジケーター追加時にコードの変更が必要
2. 共通インターフェースの欠如によるインジケーターごとの実装差異
3. インジケーター設定とUI間の連携が複雑
4. パラメーター変更のリアルタイム反映が不安定

### 解決策:

1. **インジケータープラグインシステム**:
   - 標準化されたインジケーターインターフェイス
   - プラグインとしての動的読み込みサポート
   - 設定用スキーマの標準化

2. **インジケーター設定UI**:
   - 再利用可能な設定コンポーネント
   - リアクティブな設定変更と即時反映
   - プリセット管理機能

3. **状態管理との統合**:
   - 使用中インジケーターとパラメーターのステート管理
   - ユーザー設定の永続化
   - チャート間でのインジケーター設定共有

## 色彩テーマと描画設定

現在の色彩設定は、ハードコードされた部分が多く、テーマ変更やカスタマイズが困難です。

### 課題:

1. 色設定がコンポーネント内にハードコードされている
2. テーマ切替機能が限定的
3. カスタムテーマのサポート不足
4. 描画設定の一貫性欠如

### 解決策:

1. **テーマシステムの導入**:
   - 標準化された色彩トークンを定義
   - ライト/ダークテーマの完全サポート
   - カスタムテーマのサポート

2. **描画設定の標準化**:
   - グリッド、スケール、マージンなどの設定を一元管理
   - プリセットとカスタム設定の柔軟な切り替え
   - テーマと描画設定の分離

3. **Tailwind CSS との統合**:
   - テーマ色をTailwindの変数と連携
   - コンポーネント間で一貫した色使用
   - 設定変更の即時反映

## TypeScript エラー解消計画

TypeScript のコンパイルエラーを解消し、型安全なコードベースに改善するための計画です。

### 課題:

1. バレルファイルや types/store/* で export が抜けている (TS2305)
2. UIコンポーネントやセレクターが旧フィールド名を参照している (TS2339)
3. immerSet と slice-creator の setter 型が不一致 (TS2345/2554/2322)
4. コールバック関数に型アノテーションがない (TS7006)
5. テストが削除済みストアやvitest型をインポート (TS2307)
6. 3層分割後の重複export (TS2459/2300)

### 解決策:

1. **バレル修正と型エクスポート整理**:
   - store/barrel.ts の重複export排除
   - types/store/index.ts, chart.ts, ui.ts の適切な型エクスポート
   - 型定義の階層構造の整理

2. **セレクター型安全化**:
   - store/symbol/selectors.ts と store/selectors.ts の型定義改善
   - 明示的な戻り値型の指定
   - プロパティ名の統一（favorite → isFavorite など）

3. **状態操作関数の統一**:
   - types/store/core.ts に共通 MutateDraft/SliceCreator 型を導入
   - immerSet と slice-creator の型互換性確保
   - 一貫したスライス作成パターンの適用

4. **テスト環境の整備**:
   - テスト用のパスエイリアスと vitest 型追加
   - モック用の型定義とファクトリ関数の実装
   - 不足モジュールのスタブ作成
